Meeting Notes

Call with Anirudh

July 21, 2026  ·  Anirudh (DevRev)
Reaction to the Core Idea (Sim Environments & Gyms)
  • DevRev does have a tough time creating simulation environments for third-party systems
  • But DevRev itself is very in-house and self-contained — they could just clone their own infrastructure into an alternative sandbox and test their agents there
  • As they scale up and have to talk to more third-party systems, they do see value in a co-simulation environment
  • They flagged Arga Labs (spelled "arga", from YC) as one such alternative for those environments
Biggest Input — Simulate the User, Not the Trajectory
Gym creation as currently proposed = building the ideal trajectories, which is essentially "monkey work." The more scalable, more productized move is: how well can you simulate the actual user? That sits one abstraction before the trajectory itself.
  • The pitch becomes: "I have a catalog of users specific to your domain — exactly what you need to test with"
  • By domain, the profiles are knowable up front:
    • CX — you know the different profiles of users likely to show up
    • BFSI — same; you can enumerate persona types and test against them
  • Advantage: a lot easier to package and market than hand-built trajectories
Impact of Memory in Testing (Internal Agent Flows)
  • Today Anirudh has no way of testing a lot of his internal agent flows
  • Examples of internal flows with no coverage:
    • Flows for curating memory
    • Flows for utilizing memory
    • Agent loops for curating knowledge articles
    • Agent flows for routing tickets
  • These are internal, not consumer-facing — so there is no user to simulate, but there are scenarios to simulate
  • No standardized benches exist for any of this — these scenarios could be folded into the agent construct
On the Framework / Harness Question
  • DevRev uses their own agent framework / harness
  • A lot of mature agent teams will just build it themselves
  • To sell to those teams: give them a way to create user personas on their system of record and let them figure out how to call their own endpoint
  • Caveat from Anirudh: that alone is "not really as much a solution"
Who to Talk To / GTM
  • Talk to people who really have things going on with agents and need to actually go for evals — e.g. Haiku, ClearTax (names per dictation — confirm spelling/scope)
  • Look at Galileo's customer base and go sell into that
Takeaways
  • Reframe the wedge around user simulationPosition SimGym's differentiator as a domain-specific catalog of simulated users (CX, BFSI, …), one abstraction above hand-authored trajectories — more scalable and easier to market.
  • Add "scenario" gyms for internal / memory flowsExtend beyond consumer-facing agents: memory curation, memory utilization, knowledge-article curation, ticket routing — scenarios to simulate where there's no user and no standardized bench.
  • Personas-on-your-system-of-record for mature teamsFor teams with their own harness, offer persona generation + a callable endpoint — but recognize it's a partial solution, not the whole product.
  • Build the design-partner list off Galileo's basePrioritize agent-heavy teams doing real evals (Haiku, ClearTax as leads to validate).
